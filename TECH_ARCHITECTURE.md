# 「时空纪元 (TimeMap)」技术架构设计文档

## 一、 系统整体架构
时空纪元 (TimeMap) 作为一个重前端交互、重空间数据分析的 3D 历史科普产品，整体架构分为：**强表现力的 3D 前端**、**擅长时空查询的后端服务**，以及**基于 LLM Agent 的自动化数据清洗流水线**。

## 二、 前端技术栈选型与 3D 引擎对比

### 1. 基础技术栈
*   **框架**: Vue 3 + TypeScript（响应式机制适合处理复杂的时间轴状态和 UI 面板交互）。
*   **构建工具**: Vite（极速启动，适合大型 3D 项目的模块热更新）。
*   **UI/样式**: Tailwind CSS（便于快速实现深色模式 Dark Theme 和毛玻璃拟态 Glassmorphism 效果）。
*   **状态管理**: Pinia（管理全局时间轴状态、视角经纬度、活跃的历史事件卡片等）。

### 2. 核心 3D 引擎选型对比 (Cesium.js vs Three.js)

| 维度 | Cesium.js | Three.js | 结论/适用场景 |
| :--- | :--- | :--- | :--- |
| **核心定位** | 专业级 WebGIS 引擎，专为真实地球和空间坐标设计。 | 通用 WebGL 3D 引擎，灵活性极高，社区庞大。 | 各有侧重 |
| **坐标系统** | 内置 WGS84 真实地理坐标系，完美支持经纬度到 3D 坐标的转换，天然适配地理数据。 | 默认笛卡尔坐标系，需要手动进行球面墨卡托投影和经纬度到 3D 坐标的映射计算。 | **Cesium 胜出**（历史事件强依赖真实地理坐标和多边形疆域）。|
| **宏观/微观视角切换** | 强悍的瓦片调度能力，支持从太空平滑无极缩放到街道级别。 | 需自行实现复杂的细节层级 (LOD) 和地球瓦片分块加载逻辑。 | **Cesium 胜出**（支持时空视角的宏大叙事与层级下钻）。 |
| **视觉表现与特效** | 底层材质定制较难，做极致酷炫的特效（如粒子流光）成本高，UI 往往带有浓厚的工业 GIS 味。 | 极度灵活，拥有庞大的 Shader 社区，非常适合做炫酷的流光、粒子特效、水墨晕染。 | **Three.js 胜出**（本项目对视觉设计、蝴蝶效应流光连线要求极高）。|
| **性能与体积** | 包体积大，加载庞大。 | 包体积相对较小，定制化程度高。 | **Three.js 胜出** |

**最终选型建议**：
鉴于本项目高度强调“史诗感”、“赛博蓝/辉光金的UI配色”以及“蝴蝶效应”的粒子流光特效，**建议核心采用 Three.js (搭配 custom Shaders)**。
因为产品不需要真实世界的精准高程数据（DEM）和实景卫星影像。通过提取高精度的 GeoJSON 数据，配合 Three.js 绘制低多边形（Low-poly）或暗黑风格的 3D 地球，能最大化保障视觉效果。经纬度转换可通过引入 `d3-geo` 等数学库辅助计算。

## 三、 数据存储方案 (时空与地理查询)

历史数据的两大核心维度：**空间性**（古地名的经纬度、帝国疆域的多边形边界）和**时间性**（跨越万年的时间线，包含公元前 BCE 的负数年份）。

*   **主数据库: PostgreSQL + PostGIS 插件**
    *   **空间查询**: PostGIS 是极佳的空间数据库，完美支持点（历史事件触发地）、线（丝绸之路、行军路线）、面（王朝疆界）的存储与索引。可高效执行空间分析，例如“查询公元220年坐标(x,y)属于哪个政权”、“查询某区域内爆发过的所有战役”。
    *   **时间索引**: PostgreSQL 对复杂的时间范围（Range Types）和负数年份有较好的支持，适合构建跨越千年的时间轴检索引擎。
*   **因果图谱数据库: Neo4j (针对 V3.0 蝴蝶效应)**
    *   历史事件的“直接/间接引发”关系构成了复杂的因果网。Neo4j 等图数据库非常适合存储和快速遍历这种网状的“蝴蝶效应”数据，前端调用时可迅速拉出几度人脉/事件连线。
*   **缓存与响应提速: Redis**
    *   缓存高频访问的时间片段（如二战、三国等热点时期）的事件列表和疆域 GeoJSON 数据，确保拖拽底部时间轴时，地图和数据的极速响应。

## 四、 AI 数据清洗流水线方案

为了实现“History as Code”和持续的 DLC 数据包更新，设计基于大语言模型 (LLM) 和 Agent 的自动化流水线：

1.  **数据抓取与监听 (Cron Data Collector)**
    *   后端运行定时任务，监听维基百科历史分类更新、学术文献 RSS 等半结构化/非结构化数据源。
2.  **大模型清洗与抽取 Agent (LLM Extraction Pipeline)**
    *   利用如 GPT-4 / Claude 3 等具备强推理能力的 LLM 提取文本。
    *   **工作流**：将新闻或长篇维基百科词条输入 LLM -> 提取并对齐关键要素 -> 输出标准化 JSON。
    *   **核心挑战 (古今地名映射)**：当文本提到“夷陵之战”或“君士坦丁堡”时，Agent 需调用地理逆编码工具或外部历史地理知识库，将其自动映射并转换为现代标准的经纬度坐标。
3.  **UGC 纠错与自动初审 Agent (Wiki-style Review System)**
    *   针对用户提交的“史料报错”（如某战役伤亡人数争议），触发“历史考证 Agent”。
    *   该 Agent 自动联网检索多语言学术资料、进行交叉比对，出具一份包含“史料来源链接”和“置信度打分”的审查报告。
    *   最终交由人类专家（后台管理员）进行最终把关与“一键 Merge”，合并入主干数据库。
4.  **数据 DLC 版本化发布**
    *   清洗落库的结构化数据，按历史大事件、时代主题打包为 JSON/GeoJSON 静态文件。利用 Git 进行版本控制，以 DLC 的形式推送到 CDN。前端直接拉取静态包进行渲染，极大减轻数据库压力。
